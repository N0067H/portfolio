# Pulse — 백엔드 포트폴리오

> API 엔드포인트를 등록하면 주기적으로 HTTP 헬스체크를 수행하고 결과를 기록·알림하는 모니터링 서비스  
> Java · Spring Boot · DynamoDB · SNS · EC2 · S3 · CloudFront · CloudWatch · Terraform

**GitHub**: https://github.com/n0067h/pulse-backend  
**Demo**: https://youtu.be/f4yI23oUCqU  
**기간**: 2026.02 ~ 2026.05 (개인)

---

## 1. Problem

외부 서비스 또는 내가 운영하는 API가 살아있는지 확인하려면 직접 브라우저를 열거나 curl을 쳐야 한다.  
장애가 나도 알아챌 수 없고, 이력도 남지 않는다.

→ **URL 등록 → 주기 체크 → 이력 저장 → 임계값 초과 시 알림** 파이프라인을 구축하기로 결정.

| 기능 | 설명 |
|------|------|
| 엔드포인트 관리 | URL·체크 주기·타임아웃·임계값 등록/수정/삭제 |
| 헬스체크 스케줄러 | 비동기 HTTP 요청, 타임아웃·재시도 포함 |
| 이력 저장 | 체크 결과(상태코드·응답시간) DynamoDB 기록 |
| 장애 알림 | 연속 실패 횟수가 임계값 초과 시 SNS 이메일 발송, 쿨다운으로 중복 방지 |
| 프론트엔드 | S3 + CloudFront 정적 호스팅 |

## 2. Architecture

```
┌─────────────────────────────────────────────────┐
│  CloudFront (단일 배포)                           │
│                                                 │
│  /apis/* → EC2 (캐싱 없음)                       │
│  /*      → S3 (캐싱, OAC SigV4 접근)             │
└───────────────────┬─────────────────────────────┘
                    │
         ┌──────────▼──────────────┐
         │  Spring Boot (EC2)       │
         │                         │
         │  ┌──────────────────┐   │
         │  │  REST Controller  │   │
         │  └────────┬─────────┘   │
         │           │             │
         │  ┌────────▼─────────┐   │
         │  │  CheckScheduler   │   │   ← API별 독립 ScheduledExecutorService
         │  │  (In-Memory)      │   │     앱 시작 시 1회 전체 로드 후 인메모리 스케줄
         │  └────────┬─────────┘   │
         │           │             │
         │  ┌────────▼─────────┐   │
         │  │  BlockingWorker   │   │   ← 전용 스레드 풀 (DB 저장·알림 분리)
         │  └────────┬─────────┘   │
         └───────────┼─────────────┘
                     │
        ┌────────────┼────────────────┐
        ▼            ▼                ▼
   DynamoDB        SNS            CloudWatch
 (API·이력 저장)  (이메일 알림)    (인스턴스 모니터링)
```

### 인프라 (Terraform)

- EC2, DynamoDB, SNS, S3, CloudFront, IAM 역할을 Terraform으로 프로비저닝
- 수동 콘솔 조작 없이 `terraform apply` 한 번으로 전체 환경 재현
- S3 버킷은 퍼블릭 접근 차단, OAC(SigV4)를 통해 CloudFront만 접근 허용

### 헬스체크 흐름 (Sequence)

```
CheckScheduler         HTTP Client         BlockingWorker       DynamoDB
      │                    │                     │                  │
      │── 타이머 만료 ───▶│                     │                  │
      │                    │── HTTP GET ────▶ 외부 API             │
      │                    │◀── 응답/타임아웃 ──│                  │
      │◀── CompletableFuture │                  │                  │
      │                    │                    │                  │
      │──────────────── 결과 제출 ────────────▶│                  │
      │                    │                    │── UpdateItem ──▶│
      │                    │                    │   (결과+횟수+    │
      │                    │                    │    알림시각 병합)│
      │                    │                    │◀── 완료 ─────────│
      │                    │                    │                  │
      │◀── 완료 ───────────────────────────────│                  │
      │                    │                    │                  │
      │── 자신의 인터벌로 다음 실행 재등록       │                  │
```

## 3. Technical Decisions

### DynamoDB 선택

| 항목 | DynamoDB | RDS (PostgreSQL) |
|------|----------|------------------|
| 운영 비용 | 온디맨드, 소규모 무료 티어 | EC2 상시 가동 비용 |
| 스키마 유연성 | 체크 결과 필드 추가 자유 | 마이그레이션 필요 |
| 쿼리 복잡도 | PK/SK 기반, 단순 조회에 최적 | 복잡한 집계 쿼리 가능 |

소규모 프로젝트에서 운영 비용 최소화 + 단순 키-값 패턴이 주를 이루어 DynamoDB 선택.  
단, 히스토리 집계·통계 쿼리가 복잡해지면 RDS 전환을 고려.

### S3 + CloudFront 정적 호스팅

- EC2에 정적 파일을 올리면 인스턴스 재배포 시 파일도 함께 날아감
- S3 sync + CloudFront invalidation 스크립트로 프론트엔드 배포 자동화
- 단일 CloudFront 배포에서 `/apis/*` (EC2 오리진, 캐싱 없음)과 정적 파일 (S3 오리진, TTL 캐싱) 경로 분리

### 인메모리 스케줄 vs DB 폴링

| 방식 | 읽기 비용 | 구현 복잡도 |
|------|-----------|-----------|
| 매 주기 DB 전체 Scan | O(n) 반복 | 단순 |
| 앱 시작 시 1회 로드 + 인메모리 관리 | O(1) per check | 상태 동기화 필요 |

체크 주기마다 DB를 읽는 구조는 API 수가 늘수록 RCU가 비례 증가.  
스케줄 상태를 인메모리로 관리하고 CRUD 발생 시 동기화하는 방식 선택.

## 4. Troubleshooting & Optimization

### 4-1. 매 주기 DynamoDB 전체 Scan → O(1) per check

**문제**  
- 체크 주기마다 테이블 전체를 Scan해 메모리에 올린 뒤 애플리케이션에서 필터링
- API가 10개든 1,000개든 읽기 비용은 테이블 전체 크기에 비례 → RCU 낭비

**해결**  
- 앱 시작 시 1회 전체 조회 후 각 API를 인메모리 `ScheduledExecutorService`에 독립 등록
- 체크 완료 후 자신의 인터벌 기준으로만 다음 실행을 재등록 → 이후 DB 조회 없음
- CRUD 발생 시 DB 저장과 동시에 인메모리 스케줄도 즉시 반영

**결과**: DB 읽기 비용이 API 개수와 완전히 무관한 구조로 전환 (O(n)/주기 → O(1)/체크)

---

### 4-2. 체크 1회에 분산된 2~3회 DB 쓰기 → 단일 호출 병합

**문제**  
- 체크 결과 저장 / 연속 실패 횟수 갱신 / 다음 실행 시각 갱신이 각각 별도 네트워크 호출
- API 50개 기준 한 사이클 최대 150회 쓰기가 동일 파티션에 집중 → 핫 파티션 유발 가능

**해결**  
- 갱신 필드를 인메모리에서 먼저 병합한 뒤 `UpdateItem` 단일 호출로 통합
- 다음 실행 시각도 인메모리 상태를 참조하므로 추가 조회 없음

| Metric | Before | After | 개선 |
|--------|--------|-------|------|
| DB 쓰기 / 체크 1회 | 2회 | 1회 | -50% |
| 한 사이클 최대 쓰기 (API 50개) | 150회 | 100회 | -33% |

---

### 4-3. 이력 전체 로드 후 앱 정렬·제한 → DB Query 단계로 이관

**문제**  
- 이력 조회 시 해당 API의 전체 결과를 DynamoDB에서 읽어온 뒤 정렬·페이징을 애플리케이션에서 처리
- TTL 30일 기준 최대 43,200개를 전량 읽는 구조 → 이력 누적 시 응답 지연·RCU 폭등

**해결**  
- 소트키를 ISO 8601 형식(`2026-05-01T12:00:00Z`)으로 설계 → 사전순 = 시간순 보장
- DynamoDB Query의 `ScanIndexForward=false` + `Limit=N`으로 정렬·제한을 DB 단계에서 처리

| Metric | Before | After | 개선 |
|--------|--------|-------|------|
| 이력 조회 행 수 | 최대 43,200개 | 요청 수만큼 | -99.9% |

---

### 4-4. 알림 갱신 Read-then-Write → 기존 저장 호출에 흡수

**문제**  
- 알림 발송 시각 갱신 시 전체 아이템을 먼저 읽고 수정 후 다시 쓰는 2-step 구조
- 동시 수정 경합 시 오래된 스냅샷이 최신 값을 덮어쓰는 일관성 문제 발생 가능

**해결**  
- 알림 발송 시각을 인메모리 상태에 먼저 반영 → 체크 결과 저장 호출에 통합
- 알림 전용 `UpdateItem` 호출 완전 삭제

| Metric | Before | After | 개선 |
|--------|--------|-------|------|
| 알림 경로 추가 DB 호출 | 2회 | 0회 | 기존 저장에 흡수 |

---

### 4-5. 동시 HTTP 요청 폭주 + 블로킹 I/O 스레드 점유

**문제**  
- 주기적 일괄 스케줄링 구조에서 전체 API 요청이 동시에 시작 → 커넥션 풀 한도 초과 위험
- HTTP 응답 후 실행되는 DB 저장·알림 등 블로킹 작업이 I/O 처리 스레드를 점유

**해결**  
- API별 독립 스케줄로 요청을 자연스럽게 분산 → 동시 요청 폭주 방지
- DB 저장·알림 후처리를 전용 `BlockingWorker` 스레드 풀로 분리 → I/O 스레드 즉시 반환

## 6. Retrospective

**잘 된 것**  
- 인메모리 스케줄 관리로 DynamoDB 읽기 비용을 API 수와 완전히 무관하게 만든 구조적 개선
- Terraform으로 인프라를 코드화해 환경 재현성과 배포 일관성 확보
- ISO 8601 소트키 설계로 애플리케이션 정렬 로직 없이 DB 단계에서 처리

**아쉬운 것 / 다음에 바꿀 것**  
- DynamoDB는 단순 키-값 패턴엔 좋지만, 이력 집계·통계 쿼리가 복잡해지면 RDS로 전환 필요
- 인메모리 스케줄 상태가 EC2 인스턴스 재시작 시 초기화 → 멀티 인스턴스 확장 시 DB 기반 분산 스케줄러 고려
- 알림 쿨다운 상태도 인메모리 관리 → Redis 등 외부 상태 저장소로 분리하면 다중 인스턴스 환경에서 중복 알림 방지 가능
